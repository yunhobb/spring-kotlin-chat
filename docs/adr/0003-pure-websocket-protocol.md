# ADR-0003: 순수 WebSocket + 기존 프레임 프로토콜 유지

- 상태: 확정 (2026-07-17)
- 맥락: 서버 교체(strangler-fig). 기존 앱 무수정이 원칙.

## 결정
STOMP를 쓰지 않고 **순수 WebSocket** 위에서 **기존 앱의 프레임 프로토콜을 그대로 구현**한다. seq/ack/resync 기반 신규 프로토콜(v2)은 보류.

## 근거
- 채팅 필수 의미론(전송 ack, 읽음, 재연결 누락 복구)은 STOMP의 범용 pub/sub 모델과 어긋난다. STOMP를 써도 결국 프레임 안에 자체 규격을 또 얹게 됨.
- 멀티 파드에서 STOMP는 외부 브로커(RabbitMQ 릴레이)를 강제한다. 순수 WS면 팬아웃 버스를 자유롭게 고를 수 있다([ADR-0004](0004-redis-fanout.md)).
- 계약 유지: 앱 무수정이므로 서버는 기존 프레임 규격을 **역설계해 그대로** 구현해야 한다.

## 미결 / 리스크
- **프레임 규격 미확보**: 웹엔 실시간 채널이 없어(REST 재조회 방식) 웹 관찰로는 복원 불가. 앱/구서버 소스 필요(§6-1). `TALK`/`ack`/`sync` 키워드만 확인.
- **발신자 멀티디바이스 에코 미결정**: 현재 전송 유스케이스는 발신자를 수신 대상에서 제외 → 같은 사용자의 다른 기기 실시간 미동기화. 레거시가 로컬 에코인지 재조회인지 확인 후 결정. 상세 [../realtime-and-session-design.md](../realtime-and-session-design.md) §7.
