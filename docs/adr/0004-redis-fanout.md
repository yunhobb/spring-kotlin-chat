# ADR-0004: Redis Pub/Sub 팬아웃 + 세션 레지스트리

- 상태: 확정 (2026-07-17)
- 맥락: 멀티 파드에서 "수신자 소켓을 든 파드로" 메시지를 전달해야 함. 기존 스택은 SQS + DynamoDB.

## 결정
파드 간 실시간 팬아웃과 세션 위치 추적에 **ElastiCache Redis**를 신규 도입(유일한 신규 인프라). SQS는 비동기 경로에 유지.

- Redis **Pub/Sub**: 파드 채널 `chat:pod:{podId}`로 타겟 발행.
- Redis **세션 레지스트리**: storeSeq → 파드 매핑(TTL). 타겟 라우팅 + 오프라인 판정.
- **SQS**: 오프라인 푸시(FCM/APNs), 사기 감지 연동, 배지 집계 등 유실 불가 작업.

## 근거
- **SQS는 팬아웃에 부적합**: 큐는 1소비자 모델이라 "수신자 파드 전부가 봐야 하는" 팬아웃과 반대. SNS→파드별 큐 동적 생성은 오토스케일링과 상극.
- **Kafka는 과함**: 모든 파드가 받아야 하는 실시간 홉에 컨슈머 그룹 모델·지연이 안 맞음. 내구성 파이프라인이 생기면 그때 추가.
- Redis Pub/Sub 유실 가능성은 채팅에서 문제 없음 — 원본은 DynamoDB(source of truth), 클라 재조회·재연결이 복구.

## 원칙
- 실시간 홉 = **best-effort**, 정합성 = **DB**. 실시간을 신뢰성 채널로 취급 금지.
- 페이로드 포맷은 우리 소관(레거시 계약 아님). 단 컷오버 신구 브리지와 정합 필요.
