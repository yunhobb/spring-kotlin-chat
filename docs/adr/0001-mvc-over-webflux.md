# ADR-0001: WebFlux 대신 MVC + 가상 스레드

- 상태: 확정 (2026-07-17)
- 맥락: 피크 WS 동접 5만~10만, MAU 110만→300만, EKS.

## 결정
Spring **MVC + JDK 21 가상 스레드**로 간다. WebFlux 전면 도입은 기각.

## 근거
- 이 규모에서 WebFlux의 실이득은 **커넥션 유지 레이어**뿐이다. 나머지 비즈니스 로직은 리액티브의 디버깅·온보딩 비용만 상속한다.
- 가상 스레드가 블로킹 IO(동기 DynamoDB SDK 등)의 "스레드 점유" 논거를 해소한다. 스레드-per-요청 모델을 유지하면서 확장.
- Tomcat WebSocket은 NIO 기반이라 **유휴 연결이 플랫폼 스레드를 점유하지 않는다** — 상시 연결 5만~10만을 MVC로 감당 가능.

## 대가 / 완화
- 커넥션 밀도가 극단적으로 커지면 연결 레이어만 별도 최적화 필요 → `ws-gateway` 모듈을 port 경계로 격리해 향후 Netty/WebFlux로 추출 가능하게 둠([ADR-0002](0002-modular-monolith.md)).
- 코루틴/`suspend`/Reactor 도입 금지(혼재 시 이득 없이 복잡도만 증가).
